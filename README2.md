### 1. config.py - 基础设施与模型中心
- 作用 ：
  - 环境适配 ：加载 .env 配置文件，统一管理所有 API 密钥、接口地址和模型名称。
  - 模型抽象 ：通过 get_llm() 和 get_embeddings() 方法，将不同服务商（阿里云百炼、Ollama、OpenAI 等）的调用逻辑封装。
- 双向抑制中的角色 ：
  - 参数控制 ：为「内容生成」智能体设置稍高的温度（0.3）以保证表达流畅；为「校验/检测」智能体设置零温度（0.0），确保判断的严谨性和结果的一致性。
- 联系 ：它是系统的「心脏」，被 agents.py 和 knowledge_base.py 调用，确保全链路使用统一的模型底座。

### 2. knowledge_base.py - 权威知识引擎 (RAG)
- 作用 ：
  - 知识固化 ：利用 ChromaDB 向量数据库存储教材、课标等权威文档。
  - 精准检索 ：将用户提问转化为向量，从数据库中召回最相关的「金标准」片段。
- 双向抑制中的角色 ：
  - 正向锚定 ：为生成内容提供唯一的「事实依据」。
  - 反向更新 ：支持将反向迭代智能体产生的修正内容重新存入数据库。
- 联系 ：作为系统的「记忆库」，由 workflow.py 调度，为所有智能体提供原材料。
### 3. agents.py - 多智能体大脑
- 作用 ：
  - 角色定义 ：定义了 7 个具备特定 System Prompt 的专业智能体（如权威锚定专家、幻觉检测专家等）。
  - 职责划分 ：将复杂的教育逻辑拆解为：锚定 -> 生成 -> 实时校验 -> 检测 -> 修正 -> 迭代。
- 双向抑制中的角色 ：
  - 指令注入 ：通过您设计的标准化 Prompt，强制 LLM 在特定环节执行特定的抑制动作（如「逐句比对」、「拒绝编造」）。
- 联系 ：它是系统的「执行层」，继承自 BaseAgent ，被 workflow.py 实例化并按序调用。
### 4. workflow.py - 流程调度总线
- 作用 ：
  - 全链路编排 ：按照「正向 -> 反向」的逻辑，串联起知识库和 7 个智能体。
  - 状态管理 ：传递各环节的中间结果（如：将锚定智能体的「金标准」传递给生成智能体）。
- 双向抑制中的角色 ：
  - 闭环实现 ：确保正向生成的初稿必须经过反向检测和修正后才能输出，真正实现「全链路抑制」。
- 联系 ：它是系统的「神经中枢」，向上对接 UI 界面 app.py ，向下指挥所有 src 下的模块。
### 5. app.py - 可视化交互界面
- 作用 ：
  - 用户接口 ：基于 Gradio 提供 Web 界面，让非技术用户（教师/学生）能上传文档并提问。
  - 透明化展示 ：不仅展示最终结果，还展示「幻觉检测报告」和「优化日志」，增强系统的可解释性。
- 双向抑制中的角色 ：
  - 人工反馈环 ：捕获用户的反馈信息，将其送入反向迭代链路，完成「人机协作抑制」。
- 联系 ：它是系统的「皮肤」，调用 workflow.py 来处理用户请求。
### 6. test_run.py - 自动化测试脚本
- 作用 ：
  - 功能验证 ：在不启动 Web 界面的情况下，快速验证全链路代码是否能正常跑通。
  - 实验记录 ：通过打印详细的中间日志（金标准、检测报告等），方便论文数据的收集和调试。
- 联系 ：它是系统的「体检工具」，直接调用 workflow.py 和 knowledge_base.py 。
### 总结：它们是如何协同工作的？
1. 初始化 ： app.py 启动，通过 config.py 配置模型，通过 knowledge_base.py 准备好数据库。
2. 正向抑制 ：
   - 用户提问进入 workflow.py 。
   - knowledge_base.py 检索权威片段。
   - agents.py 中的 权威锚定智能体 划定知识边界。
   - 生成智能体 在 实时校验智能体 的监督下产生初稿。
3. 反向抑制 ：
   - 检测智能体 扫描初稿，寻找事实/逻辑错误。
   - 修正智能体 根据错误报告和金标准完成精准修正。
   - 迭代智能体 记录本次幻觉类型并给出优化建议。
4. 输出与进化 ：最终合规内容展示在 app.py 界面上。若用户点击反馈，系统会再次通过 workflow.py 调用迭代智能体更新知识库。
